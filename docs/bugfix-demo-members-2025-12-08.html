BUG-009: デモユーザー権限消失修正記録

seedDemoData.ts再投入時にmembersが初期化される問題

重大度: 高 修正済み
問題概要

症状

Firestoreエラー:
FirebaseError: Missing or insufficient permissions

発生条件

  • デモ環境でシフト生成後、保存を実行
  • Phase 46でパート職員追加のためseedDemoData.tsを再実行した後

問題のコード（修正前）

const facilityData: Facility = {
  facilityId: DEMO_FACILITY_ID,
  name: DEMO_FACILITY_NAME,
  members: superAdminId ? [{
    userId: superAdminId,
    role: 'super-admin',
    grantedAt: now,
  }] : [],  // ← デモユーザーが含まれていない！
  createdAt: now,
  updatedAt: now,
};
セキュリティルールとの関係（重要な発見）:
FirestoreルールのhasRole()関数はusers.facilities配列を参照する（facilities.membersではない）。
// firestore.rules (L14-34)
function hasRole(facilityId, requiredRole) {
  let userProfile = getUserProfile();  // users/{uid}を取得
  let facilities = userProfile.facilities;  // ← ここを参照！
  return checkFacilityRole(facilities, index, facilityId, requiredRole);
}
そのため、facilities.membersだけ更新しても権限は反映されない。
修正内容

修正後のロジック（最終版）

flowchart TD A[seedDemoData.ts実行] --> B{施設が存在?} B -->|Yes| C[既存membersを取得] B -->|No| D[空配列で初期化] C --> E[既存membersをコピー] D --> E E --> F{super-admin存在?} F -->|No| G[super-adminを追加] F -->|Yes| H{デモユーザー存在?} G --> H H -->|No| I[デモユーザーをeditorで追加] H -->|Yes| J[権限をeditorに強制更新] I --> K[施設データを保存] J --> K
重要（2回目の修正）:
初回修正では「既存になければ追加」というロジックだったため、既存のデモユーザー（viewer権限）がそのまま残ってしまった。
findIndexを使用して既存ユーザーを検索し、存在する場合は権限を強制的にeditorに更新するロジックに変更。
重要（3回目の修正）:
2回目修正後も権限エラーが再発。調査の結果、セキュリティルールが参照しているのはfacilities.membersではなくusers.facilitiesであることが判明。

問題の状態:
facilities/demo-facility-001.members[].role = editor ✓
users/demo-user-fixed-uid.facilities[].role = viewer ✗

解決策: 両コレクションを同期して更新するように修正。
最終版コード: 
// 既存のmembersを取得して保持
const existingFacility = await facilityRef.get();
let existingMembers: FacilityMember[] = [];
if (existingFacility.exists) {
  existingMembers = existingFacility.data()?.members || [];
}

// super-adminを追加（既存になければ）
const members: FacilityMember[] = [...existingMembers];
if (superAdminId && !members.some(m => m.userId === superAdminId)) {
  members.push({ userId: superAdminId, role: 'super-admin', grantedAt: now });
}

// デモユーザーを追加または権限を更新
const DEMO_USER_UID = 'demo-user-fixed-uid';
const existingDemoUserIndex = members.findIndex(m => m.userId === DEMO_USER_UID);
if (existingDemoUserIndex >= 0) {
  // 既存の場合は権限をeditorに更新（viewerから変更）
  members[existingDemoUserIndex] = {
    ...members[existingDemoUserIndex],
    role: 'editor',  // Phase 43.2.1: 保存可能にするためeditor
  };
} else {
  // 新規追加
  members.push({ userId: DEMO_USER_UID, role: 'editor', grantedAt: now });
}

// 【3回目修正】users.facilitiesも同期して更新
const demoUserRef = db.collection('users').doc(DEMO_USER_UID);
const demoUserDoc = await demoUserRef.get();
if (demoUserDoc.exists) {
  const userData = demoUserDoc.data();
  if (userData?.facilities) {
    const userFacilities = userData.facilities.map((f) => {
      if (f.facilityId === DEMO_FACILITY_ID) {
        return { ...f, role: 'editor' };  // users側も同期
      }
      return f;
    });
    batch.update(demoUserRef, { facilities: userFacilities });
  }
}

学びと再発防止

重要な学び

  1. 権限エラー発生時は最初にセキュリティルールを読む: どのコレクションを参照しているかを確認することが最重要
  2. データ再投入時の副作用に注意: 既存データを上書きする際、意図しないデータ削除が起こり得る
  3. 権限問題はログに現れにくい: クライアント側の「permission denied」だけでは原因特定が困難
  4. 権限データは2箇所に存在する: users.facilities[].role（SoT）と facilities.members[].role（非正規化）の両方を同期して更新
  5. 「追加」と「更新」は別ロジック: findIndex等で既存を検索し、権限等を強制更新するロジックが必要
  6. 同じエラーが2回続いたらアプローチを変える: 修正が効かない場合は根本原因の見直しが必要

再発防止チェックリスト

  • 権限エラー時はfirestore.rulesを最初に読む
  • npx tsx scripts/verifyDemoPermissions.tsでデータ整合性を確認
  • seedDemoData.ts実行後はデモログインで動作確認
  • 権限変更時はusers.facilitiesとfacilities.membersの両方を更新
  • ユーザー追加ロジックには「既存の場合は更新」も含める

検証スクリプト

npx tsx scripts/verifyDemoPermissions.ts

このスクリプトは以下を検証します：
1. users/{demo-user-fixed-uid}.facilities[].role
2. facilities/{demo-facility-001}.members[].role
3. 両者が一致しているか（データ整合性チェック）
